# XCoder

**AI Coder Console** — это интеллектуальный терминальный ассистент для разработчиков, построенный на архитектуре Clean Architecture. Он позволяет управлять файловой системой проекта, выполнять системные команды и рефакторить код через естественный язык, используя возможности больших языковых моделей (LLM).

## 🚀 Особенности

- **Полный контроль над проектом**: Создание, чтение, обновление и удаление файлов и директорий через JSON-команды.
- **Безопасное выполнение**: Whitelist системных команд (`go`, `git`, `ls`, `rm` и др.) для предотвращения случайного повреждения системы.
- **Контекстная память**: Поддержка истории диалога с настраиваемым размером окна контекста.
- **Автоматические повторные попытки**: Устойчивость к ошибкам парсинга или сетевым сбоям API.
- **Структурированный логгинг**: Подробные логи в формате JSON для отладки и аудита действий AI.
- **Clean Architecture**: Четкое разделение на Domain, Application, Infrastructure и Presentation слои.

## 🏗 Архитектура

Проект следует принципам **Clean Architecture**:

```
├── cmd/                  # Точка входа (Composition Root)
├── internal/
│   ├── application/      # Бизнес-логика (Use Cases)
│   ├── domain/           # Сущности, интерфейсы и конфигурация
│   └── infrastructure/   # Реализация репозиториев (FS, HTTP, Exec)
├── pkg/                  # Публичные утилиты (Logger, Consts)
```

### Основные компоненты

1. **Service Layer (`internal/application/services`)**: Оркестрирует поток: Запрос пользователя -> AI -> Парсинг JSON -> Выполнение команды -> Ответ AI.
2. **AI Repository (`internal/infrastructure/http/ai`)**: Абстракция над HTTP-клиентом для общения с LLM.
3. **FS Repository (`internal/infrastructure/repository`)**: Безопасная обертка над `os` для операций с файлами.
4. **Config (`internal/domain/config`)**: Управление конфигурацией через переменные окружения.

## ⚙️ Установка и Запуск

### Требования
- Go 1.21+
- Доступ к совместимому LLM API (OpenAI-compatible endpoint)

### Переменные окружения

Создайте файл `.env` или экспортируйте переменные перед запуском:

| Переменная | Описание | По умолчанию |
|------------|----------|--------------|
| `AI_MODEL` | Идентификатор модели | `qwen3.5-397b-a17b` |
| `AI_API_URL` | URL эндпоинта чата | `http://localhost:3264/api/chat/completions` |
| `AI_TIMEOUT` | Таймаут запроса к AI | `60s` |
| `MAX_RETRIES` | Максимум попыток при ошибке | `3` |
| `CONTEXT_SIZE` | Размер истории сообщений | `20` |
| `LOG_LEVEL` | Уровень логирования | `debug` |
| `APP_ENV` | Окружение (dev/prod) | `dev` |

> **Примечание**: Файл `start.md` использовался для загрузки системного промпта. Если он отсутствует, используется промпт по умолчанию.

### Запуск

```bash
# Сборка
go build -o xcoder ./cmd/main.go

# Запуск
./xcoder
```

Или через Docker Compose:
```bash
docker-compose up --build
```

## 💡 Как это работает

1. Пользователь вводит задачу на естественном языке (например: "Создай структуру Go проекта").
2. Сервис отправляет запрос в LLM с системным промптом, предписывающим отвечать строго в формате JSON.
3. AI генерирует JSON-команду (например, `create_dir`, `exec_command`).
4. XCoder парсит ответ, валидирует его и выполняет соответствующее действие в ОС.
5. Результат выполнения возвращается в контекст AI для следующего шага.
6. Цикл повторяется до получения команды `done`.

### Пример сессии

```text
> Создай папку src и файл main.go внутри
Создаю директорию src
ok
Пишу основной файл main.go
ok
Задача выполнена успешно.
```

## 🛡 Безопасность

- Выполнение команд ограничено списком `AllowedCommands` в конфиге.
- Операции с файлами происходят относительно `ProjectRoot` (текущая рабочая директория).
- Любые ошибки выполнения команд перехватываются и возвращаются в контекст AI для коррекции.
